# Haraka — TCG買取表画像自動生成・SNS投稿プラットフォーム

## プロジェクト概要

トレーディングカードゲーム（ポケモン・ONE PIECE・遊戯王）の**買取価格データを外部スプレッドシートから自動取得し、買取表画像を生成、X（旧Twitter）へスレッド投稿**するまでの業務フローを一気通貫で自動化するWebアプリケーション。

**開発形態:** 個人開発（設計・実装・インフラ構築・運用まで一人で担当）

---

| レイヤー | 技術 |

## アーキテクチャ

### モノレポ構成（4パッケージ）

```
harakawebapp/
├── packages/shared/   … 共通型定義・定数・ユーティリティ（ランタイム依存なし）
├── packages/web/      … Next.js フロントエンド → Vercel デプロイ
├── packages/api/      … Hono REST API → Cloud Run デプロイ
└── packages/job/      … バッチジョブ処理 → Cloud Run Jobs
```

- `@haraka/shared` は**ランタイム依存ゼロ**の純粋TypeScriptパッケージとして設計。`exports` フィールドでサブパス単位のtree-shakingをサポート
- 全パッケージで `Database` 型を共有し、Supabaseクエリの**型安全性を担保**

### サービス間通信

```
[Vercel (Next.js)] ──REST──→ [Cloud Run (Hono API)] ──fork──→ [Job Process]
        │                           │                              │
        └── Supabase (直接参照) ←───┴──────────────────────────────┘
```

- フロントエンドはSSR時にSupabase直接クエリ、CSR時はAPI経由
- 長時間ジョブはAPIから `child_process.fork()` でデタッチ起動（ノンブロッキング）

## 主要機能の技術的詳細

### 1. ETLデータパイプライン（10ステージ）

Google Sheetsから3つの異なるデータソース（KECAK・Haraka DB・Spectre）を取得し、統合・変換するパイプライン。

| ステージ | 処理内容 | 技術ポイント |
|---------|---------|-------------|
| ① Run作成 | パイプライン実行レコード作成 | 全ステージの外部キーとして使用 |
| ② OAuth認証 | Google OAuth トークンリフレッシュ | リフレッシュトークンフロー実装 |
| ③④ KECAKインポート | フランチャイズ別シート取得・パース | フランチャイズ別カラムマッピング（遊戯王は7列、他は5列） |
| ⑤ DB照合マップ構築 | 3段階フォールバックマッチング | `exact` → `nameGrade` → `nameOnly` の段階的照合 |
| ⑥ カード前処理 | raw → prepared変換・価格計算 | 段階的割引率（75%/80%/85%/88%）をフランチャイズ別に適用 |
| ⑦ Spectre統合 | セカンダリソースの統合 | 異なるベンダーデータの正規化 |
| ⑧ 重複排除 | ハッシュベースDedup | 優先度: KECAK(2) > SPECTRE(1) > manual(0)、同優先度なら高価格優先 |
| ⑨ 画像ヘルスチェック | URL到達性検証 | 並行20リクエスト/タイムアウト8秒/HEAD→GETフォールバック/Cloudflare JS Challenge対応 |
| ⑩ ページ計画 | カード→ページ割当 | **FFD（First Fit Decreasing）ビンパッキングアルゴリズム** |

**技術的に注力した点:**

- **3段階ファジーマッチング**: カード名+グレード+品番の完全一致 → カード名+グレード → カード名のみ、でDB照合のヒット率を最大化
- **FFDビンパッキング**: カードをタグごとにグループ化し、サイズ降順でページ（40スロット）に充填。ページ内はサブタグ→価格順でソート
- **ルールエンジン**: 優先度付きルール（exact/contains/regex）で `isolate`（専用ページ）・`exclude`（除外）・`merge`（統合）の振る舞いを制御

### 2. 画像合成エンジン（Sharp）

テンプレート画像に対して、カード画像・価格テキスト・日付スタンプを多層合成しPNG出力。

```
テンプレート(1240×1760)
  → カード画像(8列×5行=40枚) fit:cover でリサイズ・配置
  → レアリティアイコン オーバーレイ
  → 価格テキスト SVGレンダリング（赤: 買取上限 / 青: 買取下限）
  → 空スロットはカード裏面画像で充填
  → 日付スタンプ（白文字・黒縁取り）
```

- **SVGテキストレンダリング**: `Noto Sans CJK JP` フォントで日本語価格表記（¥10,000形式）を正確に描画
- **フランチャイズ別レイアウト調整**: `layout_config` (JSONB) でピクセル単位の座標制御。遊戯王は専用のcardYDelta/priceYDeltaを適用
- **並行処理**: ページ生成 concurrency=5、画像ダウンロード concurrency=8

### 3. X（Twitter）投稿機能

OAuth 2.0 PKCE フローによる認証から、テンプレートベースのスレッド投稿まで。

- **OAuth 2.0 PKCE実装**: SHA256チャレンジ生成、10分有効のstate管理（インメモリMap + 自動クリーンアップ）、トークン自動リフレッシュ（期限600秒前）
- **テンプレートエンジン**: `{{date}}` `{{franchise}}` 等のシステム変数 + カスタム変数。`resolveTemplate()` で `{{key}}` パターンを置換
- **スレッド投稿**: ヘッダーツイート → 画像4枚/ツイートでバッチ投稿 → バナーツイート。`reply_to` でスレッドチェーン構築
- **レート制限対策**: ツイート間1.5秒スリープ
- **部分失敗リトライ**: 失敗アイテムのみ再試行、アップロード済みmedia_idは再利用

### 4. フロントエンド（Next.js 15 + React 19）

| 機能 | 実装詳細 |
|------|---------|
| **リアルタイム進捗表示** | 実行状態に応じた動的ポーリング（実行中: 2秒 / アイドル: 10秒）。`useRef` でクロージャの古いstate問題を回避 |
| **画像ビューア** | Ctrl+ホイールズーム(0.5x〜5x)、ドラッグパン、ダブルクリックトグル、キーボードショートカット（矢印/Escape/0キー） |
| **ドラッグ&ドロップ並べ替え** | `@dnd-kit` によるカード順序変更。視覚フィードバック付き（opacity変化、z-index昇格） |
| **一括ダウンロード** | `JSZip` でクライアントサイドZIP生成。進捗コールバック付き |
| **インライン編集** | onBlur保存、Enter/Escapeキーボード操作、バリデーション付きモーダルフォーム |
| **トースト通知** | ジョブ完了/失敗の自動検知・通知（15秒で自動消去） |
| **デザインシステム** | Tailwind CSS v4 カスタムテーマ変数。和風の暖色系パレット（`#f1ede7`ベース）、`Noto Serif JP` 見出しフォント |

## インフラ・DevOps

### Docker マルチステージビルド

- **ビルダーステージ**: 全ワークスペースパッケージをビルド（devDependencies含む）
- **プロダクションステージ**: `npm ci --omit=dev` で軽量化、非rootユーザー実行（uid:1001）
- **日本語フォント**: `fonts-noto-cjk` をコンテナ内にインストール（Sharp SVGレンダリング用）

### CI/CD

- **Cloud Build**: Dockerビルド → Container Registry → Cloud Runデプロイ（asia-northeast1, Memory: 1Gi, Timeout: 300s）
- **Vercel**: `shared` → `web` の順でビルド。`vercel.json` でビルドコマンドとoutputDirectoryを明示指定

### データベース設計

- **14テーブル / 13マイグレーション**で段階的にスキーマ進化
- **JSONB活用**: `raw_row`（デバッグ用生データ保持）、`layout_config`（柔軟なレイアウト座標定義）
- **複合ユニーク制約**: `db_card (franchise, card_name, grade, list_no)` でUPSERTを安全に実行
- **配列型**: `generated_page.card_ids UUID[]` でページ内カード順序を管理

## 技術的チャレンジと工夫

### 1. パフォーマンス最適化

- **並行制御**: 画像ヘルスチェック20並列、ページ生成5並列、画像DL 8並列と用途別に最適なconcurrencyを設定
- **バッチ処理**: Supabase RLS/クエリサイズ制限を考慮し500件単位でINSERT、100件単位でIN句クエリ
- **メモリ内キャッシュ**: レアリティアイコンをフランチャイズ単位でキャッシュし、同一ジョブ内の重複DLを排除

### 2. 堅牢性・障害対応

- **画像URL検証**: HEAD → GET(Range: bytes=0-0) の2段階フォールバック。Cloudflare JS Challenge (403 + `cf-mitigated`ヘッダー) を正常応答として判定
- **グレースフルデグラデーション**: BOXテンプレートDL失敗→通常テンプレートにフォールバック、レアリティアイコン欠落→ログ出力のみで生成続行
- **非同期ライトバック**: Google Sheets更新はfire-and-forget。失敗してもAPIレスポンスはブロックしない

### 3. 価格計算アルゴリズム

- GAS (Google Apps Script) v3.15.0からの移植
- `niceLowerBound()`: 価格帯別の候補ステップ（500/1000/2000/5000/10000/20000/50000）から最適な丸め値を選択。差分最小化 + 同差分時は大きいステップを優先

### 4. ページ計画アルゴリズム

- **ルール処理** → **FFDビンパッキング** → **ページ内ソート** → **ラベル生成**の4段階
- 丸数字（①②③...⑳）による重複タグラベルの自動採番

1. **設計力**: モノレポ構成で4パッケージを型安全に連携。shared パッケージによる依存ゼロの共通基盤設計

5. **インフラ**: Docker マルチステージビルド、Cloud Build CI/CD、Cloud Run + Vercel のハイブリッドデプロイ
6. **外部API連携**: Google Sheets/Drive API、X API v2（メディアアップロード・スレッド投稿）
7. **実務的な課題解決**: Cloudflare JS Challenge対応、バッチサイズ制限の考慮、グレースフルデグラデーション
